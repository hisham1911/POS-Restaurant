# مراجعة ميزة التوصيل — ملاحظات وتاسكات مطلوبة

## الهدف من هذا الملف
هذا الملف يوثق مراجعة منطقية لميزة التوصيل الحالية في النظام، ويجمع المشاكل الموجودة والتاسكات المطلوبة لتحويلها إلى ميزة مكتملة وقابلة للاستخدام فعليًا.

**مهم:**
- هذا الملف للتقييم والتخطيط فقط.
- لا يتضمن تنفيذ فعلي للتعديلات.
- الهدف هو الاتفاق على الصورة الصحيحة قبل بدء التطوير.

---

## الملخص التنفيذي
ميزة التوصيل الحالية ليست مكتملة كمنتج جاهز للاستخدام، بل هي مجموعة أجزاء منفصلة:
- صفحة لإدارة المناديب
- حقول توصيل داخل الطلب
- API لتعيين مندوب وتحديث حالة التوصيل

لكن لا يوجد Workflow كامل يربط هذه الأجزاء معًا بشكل عملي.

النتيجة الحالية:
- يمكن إنشاء مندوب
- يمكن إنشاء طلب توصيل
- توجد API لتعيين مندوب
- لكن المستخدم لا يملك رحلة استخدام كاملة لإدارة التوصيل من الواجهة

---

## ملاحظات أساسية على تصميم الميزة

### 1. نموذج الطلب الحالي غير مناسب لطبيعة المشروع
النظام الحالي يفترض منطق مطعم من خلال أنواع الطلب:
- `DineIn`
- `Takeaway`
- `Delivery`

لكن المطلوب الحالي ليس نظام مطعم.

هذا يسبب مشاكل مباشرة:
- وجود خيارات لا تخص النشاط الفعلي
- تعقيد غير مبرر في واجهة الكاشير
- Default غير مناسب (`DineIn`)
- تجربة استخدام مربكة من أول خطوة

### 2. التوصيل موجود تقنيًا لكنه غير موجود تشغيليًا
الـ backend يحتوي على:
- تعيين مندوب للطلب
- تحديث حالة التوصيل

لكن الـ frontend لا يقدم طريقة عملية للمستخدم لاستخدام هذا المنطق.

### 3. صفحة المناديب ليست نظام توصيل
صفحة `المناديب` الحالية هي صفحة بيانات أساسية فقط:
- إضافة مندوب
- تعديل مندوب
- حذف مندوب
- تفعيل/تعطيل

لكنها لا تدير الطلبات ولا تعرض سير عمل التوصيل.

### 4. تجربة المستخدم الحالية ضعيفة وغير مترابطة
المستخدم يستطيع اختيار `توصيل` داخل الـ POS، لكن لا يستطيع:
- تعيين مندوب بسهولة
- متابعة حالة الطلب
- إدارة الطلبات غير المعينة
- معرفة الطلبات لدى كل مندوب
- الوصول لصفحة تشغيل واضحة للتوصيل

---

## ما الموجود حاليًا

### Backend
- Entity للمندوب
- علاقته بالطلبات
- `DeliveryService`
- `DeliveryController`
- Endpoints للتالي:
  - عرض المناديب
  - إضافة/تعديل/حذف مندوب
  - جلب المناديب النشطين
  - تعيين مندوب لطلب
  - تحديث حالة التوصيل

### Frontend
- صفحة `DeliveryPersonsPage`
- API slice للتوصيل
- Route لصفحة المناديب
- Link في الـ navigation لصفحة المناديب
- حقول داخل الـ POS لـ:
  - نوع الطلب
  - عنوان التوصيل
  - رسوم التوصيل

## المشاكل الحالية بالتفصيل

## أولًا: مشاكل المنتج والـ UX

### 1. أنواع الطلبات غير مناسبة
الموجود الآن:
- صالة
- تيك أواي
- توصيل

المشكلة:
- هذه التصنيفات خاصة بنطاق المطاعم أكثر من كونها مناسبة لنظام كاشير عام.
- المستخدم ذكر صراحة أن المشروع ليس مطعمًا في هذه المرحلة.

### 2. لا يوجد تدفق كامل لطلب التوصيل
المطلوب منطقيًا:
1. إنشاء طلب توصيل
2. ربط عميل أو بيانات استلام
3. تعيين مندوب
4. تحديث حالة التوصيل
5. متابعة الطلب حتى التسليم

الموجود فعليًا:
- الخطوة 1 جزئيًا
- الخطوات 2 إلى 5 غير مترابطة في الواجهة

### 3. صفحة الطلبات لا تدعم إدارة التوصيل
في صفحة الطلبات الحالية يتم فقط عرض حالة التوصيل Badge.

لكن لا يوجد:
- زر تعيين مندوب
- زر تغيير مندوب
- زر تحديث حالة التوصيل
- فلترة لطلبات التوصيل
- فصل بين الطلبات غير المعينة والمعينة

### 4. شاشة تفاصيل الطلب لا تعرض بيانات التوصيل بشكل كافٍ
ينقصها عرض:
- عنوان التوصيل
- رسوم التوصيل
- المندوب المعين
- حالة التوصيل
- وقت التعيين
- وقت التسليم

### 5. لا توجد صفحة تشغيل للتوصيل
لا توجد صفحة مخصصة لإدارة طلبات التوصيل مثل:
- طلبات في انتظار التعيين
- طلبات تم تعيينها
- طلبات في الطريق
- طلبات تم توصيلها
- طلبات ملغاة

وهذا يجعل ميزة التوصيل غير قابلة للتشغيل اليومي.

### 6. صفحة المناديب ليست كافية كنقطة دخول للميزة
يوجد Link لصفحة المناديب، لكن لا يوجد Link لصفحة تشغيل فعلية للتوصيل، لأن الصفحة نفسها غير موجودة.

---

## ثانيًا: مشاكل منطقية في الـ Backend

### 1. التحقق من الفرع غير كافٍ في بعض العمليات
بعض عمليات التوصيل تعتمد على `TenantId` فقط دون تشديد كافٍ على `BranchId`.

المطلوب:
- كل قراءة وكتابة تخص التوصيل يجب أن تراعي `TenantId` و `BranchId` معًا.

### 2. يمكن تعيين مندوب غير صالح تشغيليًا
ينبغي منع:
- تعيين مندوب غير نشط
- تعيين مندوب محذوف

### 3. لا توجد قيود دورة حياة واضحة لحالة التوصيل
ينبغي ضبط قواعد مثل:
- لا يمكن الانتقال إلى `OutForDelivery` بدون مندوب
- لا يمكن الانتقال إلى `Delivered` قبل `Assigned` أو `OutForDelivery`
- لا يمكن تحديث حالة طلب ليس من نوع `Delivery`
- لا يمكن تعيين مندوب لطلب ملغي أو غير صالح

### 4. الاستجابات الخاصة بالتوصيل ناقصة
بعض Responses الخاصة بالتعيين أو تحديث الحالة تعيد DTO جزئي، وليس تمثيلًا كاملًا ومتسقًا مع بقية النظام.

### 5. لا توجد Endpoints تشغيلية كافية
ينقص وجود endpoints مخصصة مثل:
- جلب طلبات التوصيل
- جلب الطلبات غير المعينة
- جلب الطلبات حسب الحالة
- جلب الطلبات حسب المندوب

---

## ثالثًا: مشاكل في تجربة الـ POS

### 1. واجهة اختيار نوع الطلب تحتاج إعادة تصميم
بدلًا من:
- صالة
- تيك أواي
- توصيل

ينبغي اعتماد واجهة تناسب النشاط الحالي مثل:
- طلب عادي
- توصيل

### 2. بيانات التوصيل غير كافية داخل الـ POS
ينبغي عند اختيار التوصيل مراجعة التالي:
- عنوان التوصيل
- رسوم التوصيل
- العميل أو رقم الهاتف
- ملاحظات التوصيل
- إمكانية تعيين مندوب فورًا أو لاحقًا عبر شاشة مخصصة

### 3. غياب الإرشاد للمستخدم
لا توجد رسائل أو خطوات توضح:
- أن الطلب سيدخل قائمة التوصيل
- أنه يحتاج تعيين مندوب
- أين يمكن متابعة حالته بعد الإنشاء

---

## التصور الصحيح للميزة
ميزة التوصيل يجب أن تُبنى على 3 محاور واضحة:

### 1. إدارة المناديب
تشمل:
- إضافة مندوب
- تعديل بياناته
- تفعيل/تعطيل
- معرفة حالته التشغيلية

### 2. إنشاء طلب توصيل
من الـ POS أو من شاشة الطلبات، مع:
- بيانات العميل
- عنوان التوصيل
- رسوم التوصيل
- ملاحظات
- إمكانية تجهيز الطلب للديسباتش

### 3. إدارة تشغيل التوصيل
من خلال شاشة تشغيل مستقلة، تتضمن:
- قائمة الطلبات غير المعينة
- قائمة الطلبات الجاهزة أو المعينة
- تعيين/تغيير المندوب
- تحديث الحالة
- تتبع الطلب حتى التسليم

## التاسكات المطلوبة

## Phase A — إعادة ضبط المفهوم الوظيفي للتوصيل

### Task A1
مراجعة `OrderType` واختيار نموذج مناسب للنشاط الحالي بدل منطق المطعم.

### Task A2
تحديد ما إذا كان المطلوب مؤقتًا:
- إخفاء `DineIn` و `Takeaway` من الـ frontend فقط
أو:
- إعادة تصميم `OrderType` بالكامل في backend وfrontend

### Task A3
تحديد الـ default الصحيح لنوع الطلب داخل الـ POS.

---

## Phase B — تحسين إنشاء طلب التوصيل

### Task B1
إعادة تصميم قسم `نوع الطلب` في الـ POS ليعكس احتياج النشاط الفعلي.

### Task B2
فرض Validation واضح على بيانات التوصيل عند اختيار `Delivery`.

### Task B3
تحديد الحد الأدنى الإلزامي لإنشاء طلب توصيل، مثل:
- عنوان التوصيل
- عميل أو رقم هاتف
- رسوم التوصيل إذا كانت مستخدمة

### Task B4
توضيح ما يحدث بعد إنشاء طلب التوصيل برسائل UX واضحة.

---

## Phase C — تمكين ربط الطلب بالمندوب

### Task C1
إضافة UI لتعيين مندوب للطلب من مكان عملي مناسب.

### Task C2
تحديد أفضل مكان للتعيين:
- من صفحة الطلبات
- من تفاصيل الطلب
- من صفحة تشغيل التوصيل
- أو أكثر من نقطة حسب الدور

### Task C3
إظهار اسم المندوب المعين بوضوح في الطلبات والتفاصيل.

### Task C4
السماح بتغيير المندوب بشكل منطقي وآمن حسب حالة الطلب.

---

## Phase D — إدارة حالة التوصيل

### Task D1
إضافة واجهة لتحديث حالة التوصيل من الـ frontend.

### Task D2
تعريف الحالات التشغيلية بدقة، مثل:
- بانتظار التعيين
- تم التعيين
- في الطريق
- تم التوصيل
- ملغي

### Task D3
تطبيق قواعد انتقال بين الحالات بشكل صارم في backend.

### Task D4
عرض الحالة الحالية بشكل واضح ومفهوم في كل من:
- صفحة الطلبات
- تفاصيل الطلب
- صفحة تشغيل التوصيل

---

## Phase E — إنشاء شاشة تشغيل للتوصيل

### Task E1
تصميم صفحة جديدة لإدارة طلبات التوصيل.

### Task E2
تحديد Tabs أو Sections داخل الصفحة مثل:
- غير معين
- معين
- في الطريق
- تم التوصيل
- ملغي

### Task E3
إضافة Actions مباشرة من الصفحة:
- تعيين مندوب
- تغيير مندوب
- تحديث الحالة
- فتح تفاصيل الطلب

### Task E4
إضافة فلاتر مناسبة مثل:
- حسب الحالة
- حسب المندوب
- حسب التاريخ
- حسب الفرع

---

## Phase F — تحسين صفحة الطلبات والتفاصيل

### Task F2

### Task F3

### Task F4

- نوع الطلب

### Task G1

### Task G2
منع تعيين مندوب غير نشط أو محذوف.

### Task G3
منع تعيين مندوب أو تحديث حالة لطلب غير صالح.

### Task G4

### Task H2

### Task H3

## أولويات التنفيذ المقترحة

### أولوية عالية
- إزالة منطق المطعم من الواجهة الحالية أو عزله
- تمكين تعيين مندوب من الواجهة
- تمكين تحديث حالة التوصيل من الواجهة
- تقوية validation والقيود المنطقية في backend

### أولوية متوسطة
- إنشاء شاشة تشغيل كاملة للتوصيل
- تحسين عرض بيانات التوصيل في صفحة الطلبات والتفاصيل
- تحسين navigation ونقاط الوصول

### أولوية لاحقة
- إعادة تصميم أعمق لنوع الطلب على مستوى الدومين إذا تم الاتفاق على ذلك

## قرار مطلوب قبل التنفيذ
قبل بدء التطوير، يلزم حسم النقاط التالية:

### القرار 1
هل سيتم إلغاء منطق `DineIn/Takeaway` تمامًا أم فقط إخفاؤه مؤقتًا من الواجهة؟

### القرار 2
هل تعيين المندوب يتم:
- أثناء إنشاء الطلب
- بعد إنشاء الطلب
- من شاشة تشغيل مستقلة
- أم كل ما سبق حسب الصلاحيات؟

### القرار 3
ما هي الحالات التشغيلية الرسمية للتوصيل التي سيتم اعتمادها؟

### القرار 4
هل المطلوب صفحة تشغيل مستقلة للتوصيل الآن، أم نبدأ بتفعيل التعيين والحالة من صفحة الطلبات أولًا؟

---

## التوصية
أفضل مسار عمل ليس ترقيع التوصيل الحالي، بل استكماله كميزة مكتملة بالترتيب التالي:

1. ضبط المفهوم الوظيفي للتوصيل ليتناسب مع النشاط
2. إصلاح تجربة إنشاء طلب التوصيل في الـ POS
3. تمكين تعيين المندوب وتحديث الحالة من الواجهة
4. إنشاء شاشة تشغيل للتوصيل
5. تقوية الـ backend validation والـ workflow rules
