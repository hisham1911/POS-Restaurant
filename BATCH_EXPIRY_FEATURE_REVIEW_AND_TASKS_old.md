# مراجعة ميزة الباتش وتاريخ الصلاحية — ملاحظات وتاسكات مطلوبة

## الهدف من هذا الملف
هذا الملف يوثق مراجعة منطقية لميزة الباتش وتاريخ الصلاحية الحالية، ويجمع المشاكل الموجودة والتاسكات المطلوبة لتحويلها إلى ميزة تشغيلية مكتملة يمكن الاعتماد عليها يوميًا.

**مهم:**
- هذا الملف للتقييم والتخطيط فقط.
- لا يتضمن تنفيذ فعلي للتعديلات.
- الهدف هو الاتفاق على الشكل التشغيلي الصحيح قبل التطوير.

---

## الملخص التنفيذي
ميزة الباتش والصلاحية لديها أساس Backend جيد نسبيًا:
- `ProductBatch` entity
- `ProductBatchesController`
- `ProductBatchService`
- API للقراءة والإنشاء والحذف والتنبيهات
- FEFO logic في الخلفية
- `BatchExpiryAlertBanner` في الواجهة

لكن من منظور المنتج، الميزة غير مكتملة تشغيليًا لأن المستخدم لا يملك شاشة حقيقية لإدارتها.

النتيجة الحالية:
- يوجد Data Model جيد
- يوجد API جيد نسبيًا
- توجد تنبيهات قرب انتهاء الصلاحية
- لكن لا توجد Feature تشغيلية مكتملة لإدارة الباتشات

---

## ما الموجود حاليًا

### Backend
- `ProductBatch` entity
- `ProductBatchService`
- `ProductBatchesController`
- Endpoints لـ:
  - عرض كل الباتشات
  - عرض باتش محدد
  - عرض باتشات منتج معين
  - إنشاء باتش يدويًا
  - حذف باتش
  - جلب تنبيهات انتهاء الصلاحية

### Frontend
- `productBatchApi.ts`
- `BatchExpiryAlertBanner`
- ظهور الـ Banner في:
  - `POSPage`
  - `InventoryPage`

لكن لا توجد:
- صفحة لعرض الباتشات
- Route مخصص للباتشات
- Link في الـ navigation
- شاشة تشغيل فعلية لإدارة الصلاحية أو الباتشات

---

## المشاكل الحالية بالتفصيل

## أولًا: مشاكل المنتج والـ UX

### 1. الميزة موجودة كـ API وتنبيه أكثر من كونها Feature مكتملة
المستخدم حاليًا يرى تنبيهًا بقرب انتهاء الصلاحية، لكنه لا يملك:
- شاشة تعرض كل الباتشات
- تفاصيل الباتشات المتأثرة
- إجراءات تشغيلية مباشرة من التنبيه
- أدوات لاتخاذ قرار عملي

### 2. لا توجد صفحة تشغيل للباتشات
المطلوب منطقيًا في ميزة بهذا الحجم:
- شاشة تعرض الباتشات حسب المنتج / الفرع / الحالة
- فلترة حسب قريب الانتهاء / منتهي / نشط / نافد
- تفاصيل لكل باتش
- إمكانية اتخاذ إجراءات منطقية

الموجود فعليًا:
- Banner فقط
- API غير مستخدمة في UI تشغيلي

### 3. التنبيه الحالي بلا Action واضح
`BatchExpiryAlertBanner` يكتفي بعرض ملخص رقمي وإمكانية الإخفاء.

لكن ينقصه:
- زر "عرض التفاصيل"
- الانتقال لقائمة الباتشات المتأثرة
- توجيه المستخدم لما يجب فعله
- إظهار المنتجات الأكثر خطورة أولًا

### 4. رؤية الميزة مقصورة على الدور وليس على الصلاحية
الـ banner يظهر فقط للـ `Admin` و `SystemOwner` بناءً على الـ role.

هذا يخلق مشكلات مثل:
- تجاهل نموذج الصلاحيات الموجود أصلًا
- منع بعض المستخدمين التشغيليين المناسبين من رؤية التنبيهات
- ربط الرؤية بالدور بدل Permission واضحة مثل `InventoryView`

### 5. لا توجد رحلة عمل لإدارة الصلاحية المنتهية
ينقص وجود Workflow واضح مثل:
- ما الذي يحدث عند انتهاء الصلاحية؟
- هل تباع؟
- هل تُحجب؟
- هل تُراجع يدويًا؟
- من يقرر؟
- أين تُنقل أو تُعلّم كمرفوضة؟

### 6. لا توجد شفافية كافية داخل صفحات المنتجات/المخزون
الميزة لا تظهر للمستخدم بشكل كافٍ في أماكن اتخاذ القرار مثل:
- صفحة المنتج
- صفحة المخزون
- شاشة البيع عند اختيار المنتج
- شاشة التفاصيل الخاصة بالصنف

---

## ثانيًا: مشاكل منطقية في الـ Backend

### 1. إنشاء الباتش يؤثر على أكثر من كيان بدون Transaction صريحة
`CreateAsync` في `ProductBatchService` يقوم بـ:
- إنشاء باتش
- تعديل `BranchInventory`
- إضافة `StockMovement`

لكن دون Transaction صريحة.

وهذا قد يسبب عدم اتساق إذا حدث خطأ في منتصف العملية.

### 2. حذف الباتش الحالي غير مضبوط تشغيليًا
`DeleteAsync` يقوم بعمل soft delete مباشرة.

لكن لا يظهر وجود قيود واضحة تمنع الحذف إذا كان:
- للباتش كمية متبقية
- له أثر في حركات مخزون
- مرتبط ببيع أو شراء أو جرد

هذا يجعل الحذف الحالي محتمل الخطورة من منظور التشغيل.

### 3. إنشاء الباتش اليدوي يحتاج تعريفًا أوضح
الخدمة الحالية تنشئ `StockMovement` من نوع `Adjustment` مع `ReferenceType = "BatchManual"`.

هذا يثير أسئلة تشغيلية:
- متى نستخدم إنشاء باتش يدوي؟
- هل هو بديل عن فاتورة شراء؟
- هل هو فقط للتسويات؟
- كيف يتم تمييزه في التقارير؟

### 4. الميزة تحتاج قواعد أوضح للحالات التشغيلية
رغم وجود `BatchStatus`، ينقص وضوح العمليات المرتبطة بالحالات مثل:
- Active
- Expired
- Depleted

وما الذي يغيّر هذه الحالة، ومن يغيرها، وما أثرها على البيع.

---

## ثالثًا: مشاكل Information Architecture

### 1. لا يوجد مسار وصول فعلي للميزة
لا توجد صفحة، وبالتالي لا يوجد:
- Route مخصص
- Navigation item
- Entry point واضح من صفحة المخزون أو المنتج

### 2. الميزة لا تُقدَّم للمستخدم كجزء من إدارة المخزون
رغم أنها تخص المخزون مباشرة، إلا أن ظهورها الحالي متفرق وغير مترابط:
- Banner هنا
- API هناك
- بدون لوحة إدارة موحدة

---

## التصور الصحيح للميزة
ميزة الباتش والصلاحية يجب أن تُبنى على 4 أجزاء:

### 1. عرض ومتابعة الباتشات
- قائمة كاملة بالباتشات
- فلترة حسب الحالة
- فلترة حسب الفرع والمنتج

### 2. تنبيهات الصلاحية
- تنبيه ملخص
- شاشة تفصيلية
- أولويات واضحة حسب الخطورة

### 3. إجراءات تشغيلية
- مراجعة الباتش المنتهي
- منعه من البيع أو السماح به حسب إعدادات التينانت
- اتخاذ إجراء على الباتش القريب من الانتهاء
- ربط ذلك بالمخزون والحركات

### 4. تكامل مع الشراء والمخزون والبيع
- رؤية الباتش عند الشراء
- رؤية الباتش في المخزون
- رؤية تأثير FEFO في البيع
- وضوح أثر الحذف/النفاد/انتهاء الصلاحية

## التاسكات المطلوبة

## Phase A — بناء نقطة دخول واضحة للميزة

### Task A1
إنشاء صفحة فعلية لإدارة الباتشات.

### Task A2
إضافة Route واضح للصفحة.

### Task A3
إضافة Link في الـ navigation أو داخل المخزون كمدخل طبيعي للميزة.

---

## Phase B — تحسين تجربة العرض والمتابعة

### Task B1
تصميم جدول/قائمة للباتشات مع بيانات أساسية مثل:
- رقم الباتش
- المنتج
- الكمية
- تاريخ الإنتاج
- تاريخ الانتهاء
- عدد الأيام المتبقية
- الحالة
- الفرع

### Task B2
إضافة فلاتر حسب:
- المنتج
- الفرع
- الحالة
- قريب الانتهاء
- منتهي

### Task B3
إضافة صفحة أو Drawer/Modal لتفاصيل الباتش.

---

## Phase C — تحويل التنبيهات إلى Workflow فعلي

### Task C1
تحويل `BatchExpiryAlertBanner` من تنبيه فقط إلى مدخل تشغيلي.

### Task C2
إضافة Action واضح مثل:
- عرض التفاصيل
- فتح الباتشات الحرجة
- فتح قائمة العناصر القريبة من الانتهاء

### Task C3
إعادة ربط ظهور التنبيه بالصلاحية المناسبة بدل role فقط.

### Task C4
تحديد ما إذا كان إخفاء التنبيه مؤقتًا أو محفوظًا على مستوى المستخدم.

---

## Phase D — تقوية منطق الـ Backend

### Task D1
إضافة Transaction صريحة لإنشاء الباتش عند تعديل أكثر من جدول.

### Task D2
إعادة تعريف قواعد حذف الباتش بحيث لا يتم حذف باتش غير صالح للحذف تشغيليًا.

### Task D3
تحديد قواعد واضحة لإنشاء الباتش اليدوي ومتى يستخدم.

### Task D4
تعريف lifecycle واضح لـ `BatchStatus` وما يغير كل حالة.

---

## Phase E — تكامل الميزة مع بقية النظام

### Task E1
إظهار الباتشات في صفحة المخزون أو تفاصيل المنتج بشكل أوضح.

### Task E2
توضيح أثر FEFO للمستخدم في الواجهة.

### Task E3
إظهار الباتش المختار أو المستهلك عند نقاط البيع أو في سجل الحركات حيث يلزم.

### Task E4
ربط الصلاحية المنتهية بسياسة التينانت بشكل واضح للمستخدم الإداري.

---

## Phase F — ضبط الصلاحيات والوصول

### Task F1
تحديد من يمكنه:
- رؤية الباتشات
- إنشاء باتش يدوي
- حذف باتش
- رؤية التنبيهات

### Task F2
ربط هذه العمليات بالصلاحيات بدل الاعتماد الجزئي على الـ role.

---

## أولويات التنفيذ المقترحة

### أولوية عالية
- إنشاء شاشة فعلية للباتشات
- تحويل التنبيهات إلى workflow حقيقي
- تقوية `CreateAsync` و `DeleteAsync` تشغيليًا

### أولوية متوسطة
- ربط الميزة بصفحات المخزون والمنتجات
- توضيح FEFO والحالات التشغيلية

### أولوية لاحقة
- تحسينات أعمق في العرض والتحليلات والسياسات التفصيلية

---

## قرار مطلوب قبل التنفيذ

### القرار 1
هل إدارة الباتشات ستكون صفحة مستقلة أم جزءًا من صفحة المخزون؟

### القرار 2
هل إنشاء الباتش اليدوي مطلوب كاستخدام يومي أم كاستثناء إداري فقط؟

### القرار 3
ما هي قواعد حذف الباتش المقبولة تشغيليًا؟

### القرار 4
من هم المستخدمون الذين يجب أن يروا تنبيهات الصلاحية؟

### القرار 5
هل المطلوب فقط التنبيه، أم إدارة كاملة للباتشات المنتهية والقريبة من الانتهاء؟

---

## التوصية
أفضل مسار ليس الاكتفاء بالـ Banner الحالي، بل استكمال الميزة كأداة تشغيل مخزني حقيقية:

1. إنشاء شاشة إدارة للباتشات
2. تحويل التنبيه إلى Workflow قابل للتنفيذ
3. ضبط قواعد الإنشاء والحذف والحالات
4. ربط الميزة بالمخزون والمنتجات والبيع بشكل أوضح

---

**الميزة الحالية قوية على مستوى الداتا والـ API، لكنها ناقصة جدًا على مستوى التشغيل والواجهة ونقاط الوصول.**
